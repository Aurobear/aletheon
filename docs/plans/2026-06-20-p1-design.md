## 1. Checkpoint/Rewind System

**Source:** Reasonix `internal/checkpoint/`

### Problem

No safety net for agent file edits. If the agent makes a bad write, there's no way to roll back without manual git operations. The user loses both code state and conversation context.

### Design

Snapshot-based (not git-based). One checkpoint per user turn. Uses a `Previewer` trait to capture file state before edits.

```
User turn starts
  → Open checkpoint
  → Tool calls execute (edit tools snapshot via Previewer)
  → Turn ends
  → Checkpoint sealed with turn number + msg index

Rewind:
  → Restore file snapshots from checkpoint N
  → Truncate conversation to checkpoint N's msg index
```

**New types:**

```rust
// crates/aletheon-runtime/src/core/checkpoint.rs

/// A snapshot of a file before it was modified.
pub struct FileSnap {
    pub path: PathBuf,
    pub content: Option<String>,  // None = file didn't exist (restore deletes it)
}

/// One checkpoint per user turn.
pub struct Checkpoint {
    pub turn: usize,
    pub time: SystemTime,
    pub prompt: String,
    pub msg_index: usize,  // conversation rewind boundary
    pub files: Vec<FileSnap>,
}

/// Scope of rewind operation.
pub enum RewindScope {
    Code,          // restore file snapshots only
    Conversation,  // truncate messages only
    Both,          // both code and conversation
}

/// Checkpoint store.
pub struct CheckpointStore {
    session_dir: PathBuf,
    checkpoints: Vec<Checkpoint>,
}
```

**Previewer trait (in aletheon-abi):**

```rust
// crates/aletheon-abi/src/tool.rs

/// Tools that can preview their change without touching disk.
pub trait Previewer: Tool {
    fn preview(&self, args: &serde_json::Value) -> Option<FileSnap>;
}
```

Only edit/write tools implement `Previewer`. Bash does not (too complex to predict).

**Controller integration:**

```rust
impl Controller {
    pub fn rewind(&self, turn: usize, scope: RewindScope) -> Result<()> { ... }
    pub fn checkpoints(&self) -> &[Checkpoint] { ... }
}
```

## 2. Tool Parallelism Partitioning

**Source:** Reasonix `internal/agent/executeBatch`

### Problem

All tool calls execute sequentially. When the LLM requests multiple read-only tools (read_file, glob, grep), they could run in parallel for faster response.

### Design

Partition tool calls into contiguous batches:
- **Read-only batch** → run in parallel (max 8 concurrent)
- **Writer batch** → run serially

```rust
// crates/aletheon-runtime/src/core/react_loop.rs

/// Partition tool calls into read-only (parallel) and writer (serial) batches.
fn partition_tool_calls(calls: &[(String, String, serde_json::Value)], tools: &ToolRegistry) -> Vec<ToolBatch> {
    let mut batches = Vec::new();
    let mut current_batch = ToolBatch::Serial(Vec::new());

    for call in calls {
        let tool = tools.get(&call.1);
        let is_read_only = tool.map(|t| t.concurrency_class() == ConcurrencyClass::ReadOnly).unwrap_or(false);

        if is_read_only {
            match &mut current_batch {
                ToolBatch::Parallel(v) => v.push(call.clone()),
                ToolBatch::Serial(v) if v.is_empty() => {
                    current_batch = ToolBatch::Parallel(vec![call.clone()]);
                }
                ToolBatch::Serial(v) => {
                    batches.push(std::mem::replace(&mut current_batch, ToolBatch::Parallel(vec![call.clone()])));
                }
            }
        } else {
            match &mut current_batch {
                ToolBatch::Serial(v) => v.push(call.clone()),
                ToolBatch::Parallel(v) => {
                    batches.push(std::mem::replace(&mut current_batch, ToolBatch::Serial(vec![call.clone()])));
                }
            }
        }
    }
    batches.push(current_batch);
    batches
}

enum ToolBatch {
    Parallel(Vec<(String, String, serde_json::Value)>),
    Serial(Vec<(String, String, serde_json::Value)>),
}
```

**Execution:**

```rust
for batch in batches {
    match batch {
        ToolBatch::Parallel(calls) => {
            let futures = calls.iter().map(|(id, name, input)| execute_tool(id, name, input));
            let results = futures::future::join_all(futures).await;
            for result in results { ... }
        }
        ToolBatch::Serial(calls) => {
            for (id, name, input) in calls {
                execute_tool(id, name, input).await;
            }
        }
    }
}
```

The `ConcurrencyClass` enum already exists on the `Tool` trait (ReadOnly, Write, SideEffect).

## 3. Storm Breaker Anti-Loop

**Source:** Reasonix `internal/agent/compact.go`

### Problem

Models sometimes get stuck in loops: repeating the same failing tool call, or repeating the same successful write. This wastes tokens and time.

### Design

Detect repeated identical failures/successes and inject a directive to change approach.

```rust
// crates/aletheon-runtime/src/core/storm_breaker.rs

use std::collections::HashMap;

/// Tracks tool call patterns to detect loops.
pub struct StormBreaker {
    /// Key: (tool_name, error_signature), Value: consecutive count
    failure_counts: HashMap<(String, String), usize>,
    /// Key: tool_name, Value: consecutive success count
    success_counts: HashMap<String, usize>,
    /// Threshold to trigger
    threshold: usize,
}

impl StormBreaker {
    pub fn new(threshold: usize) -> Self { ... }

    /// Record a tool call result. Returns a directive if loop detected.
    pub fn record(&mut self, tool_name: &str, is_error: bool, content: &str) -> Option<String> {
        if is_error {
            let error_sig = self.extract_error_signature(content);
            let key = (tool_name.to_string(), error_sig);
            let count = self.failure_counts.entry(key).or_insert(0);
            *count += 1;
            if *count >= self.threshold {
                return Some(format!(
                    "⚠️ Storm breaker: {} has failed {} times with the same error. \
                     Try a completely different approach.",
                    tool_name, count
                ));
            }
            // Reset success counter
            self.success_counts.remove(tool_name);
        } else {
            let count = self.success_counts.entry(tool_name.to_string()).or_insert(0);
            *count += 1;
            if *count >= self.threshold {
                return Some(format!(
                    "⚠️ Storm breaker: {} has succeeded {} times in a row. \
                     Verify the result before continuing.",
                    tool_name, count
                ));
            }
            // Reset failure counters for this tool
            self.failure_counts.retain(|k, _| k.0 != tool_name);
        }
        None
    }

    /// Reset all counters (called at turn start).
    pub fn reset(&mut self) { ... }

    fn extract_error_signature(&self, content: &str) -> String {
        // Normalize error: take first 100 chars, lowercase, strip numbers
        content.chars().take(100).collect::<String>().to_lowercase()
    }
}
```

**Integration in ReActLoop:**

```rust
// In the tool execution loop:
if let Some(directive) = storm_breaker.record(name, is_error, &content) {
    // Inject directive as a system message
    self.messages.push(Message::user(&directive));
}
```

## Validation Criteria

### Checkpoint
- [ ] FileSnap captures file content before edit
- [ ] Checkpoint stores turn number + msg_index
- [ ] RewindScope::Code restores file snapshots
- [ ] RewindScope::Conversation truncates messages
- [ ] RewindScope::Both does both

### Tool Parallelism
- [ ] partition_tool_calls groups contiguous read-only tools
- [ ] Read-only batches execute in parallel (max 8)
- [ ] Writer batches execute serially
- [ ] Mixed tool calls produce correct batches

### Storm Breaker
- [ ] 3 consecutive identical failures triggers directive
- [ ] 3 consecutive identical successes triggers warning
- [ ] Reset clears all counters
- [ ] Error signature normalization works
